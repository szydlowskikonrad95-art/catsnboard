Co się dzieje, gdy coś pójdzie nie tak

Automat pracuje sam, bez opieki. Dlatego każda możliwa awaria ma zaplanowaną reakcję — system nie umiera od jednej usterki i nigdy nie kasuje danych klienta przez pomyłkę.

🖼️ Zdjęcie się nie pobierze
Wydarzenie wchodzi bez zdjęcia (placeholder) i nie wywala cyklu. Automat próbuje dobrać foto w kolejnych cyklach.
samo-naprawa zdjęć
📡 Źródło nie odpowiada
Spróbuje w następnym cyklu, automat żyje dalej. Porażki są liczone.
0️⃣ Źródło zwróci 0 wydarzeń
NIE czyści strony. Traktuje to jak awarię źródła, nie jak „wszystko usunięte" — dane zostają.
🔁 Ten sam import dwa razy
Zero duplikatów. Automat rozpoznaje wydarzenie po jego ID u źródła i nie dodaje drugi raz.
dedup: _pnb_source_id
⛔ Dwa cykle naraz
Blokada (lock). Tylko jeden import w danej chwili — drugi grzecznie kończy, nie gryzą się o te same dane.
💥 Źródło pada w kółko (ban/500)
Bezpiecznik (circuit breaker) — po serii porażek czeka coraz dłużej (10→20→40, max 120 min), nie łomocze martwego źródła.
☠️ Jedno wydarzenie pada 5×
Odkłada je na bok (dead letter) — przestaje próbować w kółko, resztę importuje normalnie. Widoczne w panelu.
📉 Źródło nagle dało 22 → 2
Alert + pauza wygasania. Podejrzany spadek = możliwa zmiana API — nie kasujemy, tylko ostrzegamy.
✏️ Klient ręcznie poprawił wydarzenie
Automat już go nie nadpisuje. Ręczne zmiany są chronione — źródło ich nie zamaże.
flaga: _pnb_locked
✅ Wszystko potwierdzone testami wykonaniem — nie teorią. Trzy nocne testy od zera (świeży WordPress, droga klienta) przeszły czysto: automat sam scrapuje, tłumaczy i dodaje zdjęcia, zero duplikatów, zero błędów.
